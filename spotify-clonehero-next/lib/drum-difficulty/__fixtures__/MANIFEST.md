# Drum difficulty-reduction parity fixtures

Anonymized reference charts for verifying the TypeScript ports of the two
Python drum-difficulty-reduction algorithms (HOPCAT / C3toolbox and Onyx /
Reductions.hs) reproduce the Python reducers tick-exactly. See plan
`0071-difficulties-comparison-page.md` §5.

## Provenance & anonymization

Each fixture is a **minimal, anonymized** Rock Band-style `notes.mid` derived
from a real Expert pro-drums chart, carrying only the data the two reducers
consume:

- **track 0** — generic name (`reduction-NN`) + every `set_tempo` and
  `time_signature` meta event (absolute positions preserved).
- **`PART DRUMS`** — every note event (ticks/pitches/durations byte-for-byte
  as the original) plus only `[mix … drums…]` disco/audio text markers.
- **`EVENTS`** — section markers at their **original tick positions** but with
  **generic renumbered names** (`[section sec_001]`, …). Only the boundary
  position matters to either reducer, never the section name.

Everything identifying — the sequence/song-title track name, `PART VOCALS`
lyrics, `VENUE`, copyright, and all other instrument tracks — is dropped
entirely. `song.ini` carries only placeholder `name`/`artist`/`charter`
fields. No audio is included. Fixture IDs are frozen sequential anonymous IDs
(`reduction-01`…); per project convention they are never renumbered or reused.

## How the expected outputs were produced

`expected-hopcat.json` and `expected-onyx.json` are the Python reducers'
output run **on the committed anonymized `notes.mid`** (not the original), via
`reduce_port.reduce_5lane_drums` (HOPCAT, `DEFAULT_CONFIG`) and
`onyx_reduce.drums_complete` (Onyx). Generation asserted that the anonymized
file reproduces the _original_ file's reducer output exactly for both tools,
so anonymization changed no note/timing/marker data. Each JSON is
`{hard,medium,easy: […]}`:

- HOPCAT rows: `{tick, pitch, lane}` (pitch = raw tier gem pitch).
- Onyx rows: `{tick, lane, kind, color, protype}` (resolved Pro gem; `tick`
  = exact `beats × 480`, always integral). Ticks-per-beat is 480 for every
  fixture.

### `expected-ours.json` (v5 "591ab4a" packed-binary model)

The third reducer, **Ours**, is the trained v5 (`591ab4a`, packed-binary
artifact) GBM (plan §3b/§3b-v3/§3b-v4/§3b-v5) — the SAME trained trees as v4
(`model_cache_key` unchanged, `08b9f91735e6dcf7`), repackaged from a ~36MB
pure-JSON tree dump into a ~1.9MB packed-binary format (custom byte-level
tree evaluator, no ONNX — see `model.ts`'s doc comment) with an unchanged
decode pipeline. Unlike HOPCAT/Onyx it is **not** a keep/drop-only reducer: per tier it
runs a binary **survive head** (`sigmoid(raw)`), then **survive-pool**
(replace each note's probability with the MEAN probability across every
instance of its `(groove_cluster, tick_in_measure, lane)`, for notes in a
repeated-groove measure only — a free variance reduction, computed BEFORE
thresholding), then thresholds (`≥ 0.5`), then **family-NMS** (greedy
non-max-suppression among surviving cymbal/tom notes by the POOLED survive
probability — gaps `{hard: off, medium: 180ms, easy: 250ms}` — this runs
BEFORE relane in v4, the reverse of v3's order), then per-family **relane
heads** that may move a surviving cymbal (`hihat/open-hat/crash/ride`) or tom
(`high-tom/mid-tom/floor-tom`) note to a different lane within its family (kick/snare/other are
fixed — survive-only, never relaned), then **relane-pool** (override each
family note's relaned lane with the CONFIDENCE-WEIGHTED MODAL lane across
every instance of its `(groove_cluster, tick_in_measure, source_lane)` — every
family row in a cluster casts a confidence-weighted vote regardless of
survive status), then a **chord-merge dedup** (surviving family notes that
land on the same `(ms, family, final_lane)` collapse to the highest-confidence
one), then **canonicalization** (unchanged from v3 — every instance of a
repeated Expert-chart groove is forced to the reducer's own modal reduction
for that groove; pooling already does most of this work upstream, so
canonicalize now corrects far less, but it's still required for a guaranteed
zero post-canon inconsistency). **Ours never changes a note's time** — a
surviving, non-canonicalized note keeps its original tick; only its lane may
change — so every row's `tick` is `round(beats × 480)` (matching Onyx's basis;
ticks-per-beat is 480 for every fixture). A canonicalized "donor-copied" note
(rare) gets a `tick` reconstructed the same way from its rewritten ms.

Generated by running the **real** Python decode pipeline on the committed
anonymized `notes.mid`'s already-parsed chart JSON (scan-chart parse via
`parse_all_charts.ts`, the corpus's own parser) → `prepare.py`'s pinned
`_extract_song_features_v2` (chart-only base features + `chord_has_<lane>` +
`family`) → `train_pinned_591ab4a.build_matrix_aug` (`AUG_FEATS` v7, three
absolute-position columns dropped — unchanged from v3) → the **shipped v5
packed-binary models** in `public/models/drum-difficulty/v5/` (`survive_*.bin`
/ `relane_*.bin`, read with a plain-Python re-implementation of
`pack_model.py`'s `rebin`/`traverse_packed`) → the same
`train_pinned_591ab4a` decode helpers `export_model_591ab4a.json_reduce_song`
uses (survive-pool → family-NMS → relane-pool → chord-merge) →
`consistency_metric.canonicalize` (unchanged from v4). Output was verified
note-for-note against that reference decode for all 20 fixtures × 3 tiers.

All 20 fixtures are tick-exact identical to the pre-packing v4 (pure-JSON,
fp64) fixtures, INCLUDING `reduction-15`/hard, which briefly (2026-07-22, in
between two v5 regenerations) showed a one-note difference: a kick note's
pooled survive probability was `0.49999987` under the fp64 JSON model vs
`0.50000495` under the fp16-packed model — a ~5e-6 gap, an order of magnitude
below the packed format's own documented noise floor (~7.4e-5, see
`PRODUCT_INTEGRATION.md` §8), straddling the `0.5` threshold by chance.
Switching `model.ts`'s sigmoid/softmax from `Math.exp` to a fixed portable
`exp` (`portableExp.ts`, ported from the unrelated `drum-reducer-reference`
project's `portable_exp.{py,js}` — see below) shifted the computed proba
enough to land back on the correct side of the threshold, so this fixture is
tick-exact again. This was incidental, not the point of that change (the
point is browser-to-browser determinism, not this specific fp16 boundary
case — see `portableExp.ts`'s doc comment) — the underlying fp16
quantization noise this session diagnosed is real and could reappear on a
future fixture regeneration; if it does, it's expected, not a TS-port bug.

**`drum-reducer-reference` cross-check (2026-07-22):** a newly-released,
unrelated sibling project (`~/projects/drum-to-chart/drum-reducer-reference`)
ships its own from-scratch Python + JS reference implementation of this same
`591ab4a` model (confirmed byte-identical `.bin`/`manifest.json`/
`feature_names.json` via `shasum`), with a formal `DETERMINISM_CONTRACT.md`
specifying exact tie-break rules for every decode step that can compare two
notes as equal (FAMILY-NMS ordering, relane-pool's modal vote, chord-merge
dedup, canonicalize's modal reduction). Investigating whether to adopt those
rules here found that they do **NOT** describe this deployed model's actual
decode script (`train_pinned_591ab4a.py`/`consistency_metric.py`, which this
project's fixtures are ground-truthed against) — confirmed empirically:
swapping `reduce.ts`/`consistencyMetric.ts`'s tie-breaks to match that
contract regressed 2 of the 20 real-song fixtures (extra cymbal notes
surviving that the real script drops). Reading the actual Python source
confirmed why: `apply_family_nms`'s Python `list.sort` is stable (ties keep
row-index order, not `(ms, lane_index)`); `groove_pool_lane`'s `max(...)`
picks the first max in dict-insertion order, not lowest `lane_index`;
`cand_from_predictions`'s chord-merge sort is likewise stable on row-index
order; and `consistency_metric._modal_reduction` uses `Counter.most_common`
(first-encountered wins ties), not a lexicographic tuple-list comparison.
All four changes were reverted — `reduce.ts`/`consistencyMetric.ts`'s
existing insertion-order/stable-sort tie-breaks were already correct ports
of the real script, and are the ones a future contributor should trust over
that other project's (differently-designed, equally valid for ITS OWN
purposes) documented contract. `portableExp.ts` was kept — it's justified
independently (browser-to-browser `Math.exp` non-portability), doesn't
depend on matching any tie-break convention, and is itself verified
bit-exact against the actual Python `portable_exp.py` module over 8000+
grid values (`ours/__tests__/portableExp.test.ts`).

Each JSON is `{hard,medium,easy: […]}` plus a `_meta` diagnostic key:

- Tier rows: `{tick, lane}`. **Assert on `(tick, lane)` only** — v5 fixtures
  carry no per-note diagnostic fields (`originalLane`/`family`/`relaned`/
  `confidence` aren't well-defined for canonicalize()'s donor-copied notes,
  which may be copied from a different measure instance's candidate).
- `_meta.counts`: per-tier `{nExpert, nPreCanon, nKept}` (`nPreCanon` is the
  count before canonicalization; `nKept` is the final, post-canonicalization
  count — they differ only on fixtures with repeated Expert grooves).

Two things a TS-port author should know:

- **Ours reduces far less aggressively than HOPCAT/Onyx.** Its survive head uses
  _family-level_ survival labels for cymbal/tom (a note survives if GT has any
  same-family note at that tick), so it retains a meaningfully larger fraction
  of Expert notes than HOPCAT/Onyx's much lower rates — this is the model's
  design, not an error.
- **`nExpert` counts double-kick notes.** The Ours featurizer flattens
  scan-chart's Expert notes _including_ `doubleKick` gems, so its `nExpert`
  exceeds the "Expert" column below on 2x-kick charts (e.g. reduction-14:
  6104 vs 5529 = 575 double-kick). The survive head drops most of them (GT
  tiers have no note there). The TS `featurize` port must likewise include
  double-kick notes as input rows or the vectors misalign.

## Coverage summary

- 20 fixtures, all 480 ticks-per-beat.
- Disco-flip charts (`[mix N drums*d]` markers): 5.
- Odd-meter (non-4/4 signature present): 10.
- Multiple time-signature events: 10.
- Drum rolls (pitch 126) present: 8.
- Cymbal swells (pitch 127) present: 5.
- Overdrive/star-power phrases (pitch 116): all fixtures.
- `[section …]` markers: all fixtures (the source corpus has no section-less charts).
- Note-density spread: from ~150 to ~5500 Expert gems.
